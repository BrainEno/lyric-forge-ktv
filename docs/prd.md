一、PRD

1. 产品名称

LyricForge KTV（暂名）

2. 产品定位

一款面向个人创作者、翻唱爱好者和小型工作室的本地 KTV 制作工具。
用户导入本地音频后，软件完成以下流程：

识别和预处理音频 提取伴奏与人声 对人声做本地语音识别
将切片文本拼接为完整歌词草稿 生成可编辑歌词时间轴 输出可播放的本地 KTV 工程

这个定位是我基于你要做“本地 Whisper + 伴奏提取 + 歌词整理 + 本地
KTV”这条链路做的产品收口，不依赖在线曲库，不承诺首版全自动完美歌词。

3. 产品目标 3.1 业务目标

首版解决三个核心痛点：

冷门歌或本地音频没有现成歌词 普通用户不会手工做 KTV 工程
本地离线处理比在线上传更私密，也更适合版权敏感内容 3.2 用户目标

用户希望在一台电脑上完成：

导入一首歌 一键生成伴奏和歌词草稿 手动改少量错词 导出可唱的 KTV 工程
在桌面或手机上播放和练唱 4. 目标用户 核心用户 喜欢翻唱、练歌、做伴奏的个人用户
做冷门歌曲整理的音乐爱好者 需要处理本地音频、注重隐私的创作者 次级用户
小型直播/翻唱工作室 做粤语、日语、方言歌曲整理的垂类用户 5. 非目标

首版不做这些：

在线曲库和版权采购 云端协作 自动完美校准到逐字级商用 KTV 水准
手机端完整本地伴奏分离 多人点歌厅系统 6. 核心使用场景 场景 A：桌面端制作

用户在桌面端导入一首本地歌曲，系统完成预处理、分离、转写、歌词拼接，用户在编辑器里做少量修改，导出工程。

场景 B：手机端播放与校对

用户把工程同步到手机，进行伴奏播放、歌词滚动查看、简单时间偏移调整和错词修正。

场景 C：冷门歌补全

用户导入一个没有歌词的音频，系统先给出“歌词草稿 +
时间轴”，而不是承诺“完全正确歌词”。

7. MVP 范围 7.1 桌面端必须有 本地音频导入 音频转码与标准化 人声/伴奏分离
   人声音轨本地 ASR 歌词切片合并与去重 LRC 级别时间轴编辑 KTV 播放器：伴奏 +
   滚动歌词 工程目录保存/重新打开 7.2 手机端 MVP 打开桌面导出的工程 播放伴奏
   展示滚动歌词 简单时间偏移微调 保存校对结果
8. 关键功能需求 8.1 工程管理 新建工程 导入单首音频 自动生成工程目录 最近项目列表
   崩溃后恢复未完成任务 8.2 音频处理 支持 mp3 / flac / wav / m4a
   自动转为内部统一格式 支持桌面端人声/伴奏分离 分离结果可试听和重跑 8.3
   歌词识别 对分离后人声音轨做 Whisper 识别 支持按切片识别并合并
   支持语言提示、初始提示、专有词典 输出逐行文本和时间戳草稿 8.4 歌词编辑
   切片合并 去重 断句与分段 手动修词 手动调节行时间 导出 .lrc 8.5 KTV 播放
   播放伴奏 歌词滚动高亮 拖动进度条时歌词同步定位 可调节全局歌词偏移
9. 非功能需求 桌面端离线可用 本地工程打开时间可控 错误可恢复，日志可导出
   对大文件处理要有进度条和取消按钮 工程文件结构可迁移，可从桌面复制到手机
10. 建议技术架构

我建议你用：

Flutter UI + Rust Core + FFmpeg + whisper.cpp + 桌面端可替换的伴奏分离后端

这是产品方案，不是官方固定答案。 之所以这么定，是因为 Flutter
天然覆盖桌面和手机；而重计算、音频处理、模型调用更适合放在 Rust 核心里。Flutter
官方支持桌面和移动平台；OpenCode/Ollama
只是你的开发代理组合，不是产品运行时本体。

11. 里程碑 Phase 0：可行性验证（1 周） 单首音频导入 whisper.cpp 跑通
    输出歌词草稿 播放伴奏占位版 Phase 1：桌面 MVP（3–5 周） 完整工程目录 分离 +
    转写 + LRC 编辑 基础 KTV 播放器 Phase 2：手机消费端（2–4 周） 打开工程
    播放与歌词滚动 微调时间轴 Phase 3：增强（后续） 词级时间对齐 质量评分
    多语言/方言优化 更好的伴奏后处理
12. 验收标准

首版上线前至少满足：

任意导入一首常见格式音频，能生成可再次打开的工程
能导出一份可播放的伴奏和一份可编辑 LRC 用户在 10 分钟内完成从导入到初次可唱
对单首歌的失败点可定位到“导入 / 分离 / 转写 / 拼接 / 导出”中的具体阶段 13.
核心风险 自动歌词准确率受音质、混响、语言和咬字影响很大
伴奏分离质量不稳定，可能残留人声 手机端本地大模型/分离推理成本太高
如果首版目标过大，很容易拖死节奏 二、OpenCode 的 harness
engineering：我建议你这样理解

我这里把 harness engineering 定义为：

不是只写
prompt，而是把“规则、上下文、权限、验证、交付格式”一起工程化，变成可重复的开发支架。

在 OpenCode 里，这件事最好拆成四层：

第 1 层：全局配置

放你长期不变的偏好，比如：

默认 provider 默认 model / small_model 权限策略 通用快捷键和输出习惯

OpenCode 支持全局配置文件，位置是
~/.config/opencode/opencode.json；配置会和项目级配置合并，而不是单纯覆盖。它也支持
small_model 用于轻量任务。

第 2 层：项目配置

放每个仓库不同的内容，比如：

这个项目用哪种 provider 这个项目的测试命令 这个项目是否允许自动编辑 这个项目额外
instructions 文件

OpenCode 文档说明项目根目录也可以放 opencode.json，而且它会覆盖冲突项。

第 3 层：AGENTS.md

这是最关键的一层。 OpenCode 支持通过 AGENTS.md
注入项目规则，这些规则会被带进模型上下文。

这里应该写：

项目目标 目录结构 架构约束 命名规则 测试命令 不允许触碰的区域 提交格式 回答格式
第 4 层：单次任务单

也就是你每次给 OpenCode 的任务 brief。 只写这一轮的：

目标 输入 输出 验收标准 禁止事项

不要把长期规则和项目背景重复塞进每次 prompt。 长期规则写进
AGENTS.md，任务单只写变化部分。
